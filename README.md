# DeepSeek V3.2: DSA (DeepSeek Sparse Attention)

A unofficial PyTorch implementation of **DeepSeek V3.2**, which
extends DeepSeek V3 (MLA + MoE) with **DeepSeek Sparse Attention (DSA)**: a
Lightning Indexer that selects only the top-k most relevant KV entries per query
token, reducing attention complexity from O(L^2) to O(L * k)

NOTE: This README document was mostly generated by AI (used claude) (it does a better job then me at formatting)

NOTE: Obviously I could not finetune or work with large models thus small experiments were used as proof

## What is new in V3.2? Mainly DeepSeek Sparse Attention


### Indexer equation (paper Eq. 1)


$I_{t,s} = \sum_{j=1}^{H_I}  w^I_{t,j} . ReLU(q^I_{t,j} . k^I_s)$

| Symbol | Description | Paper (671B) default |
|--------|-------------|---------------|
| H_I | Indexer heads | 64 |
| D_I | Indexer dimension | 128 |
| k | Top-k tokens selected | 2 048 |
| q^I, k^I | Projected from hidden states h_t / h_s, RoPE-applied | |
| w^I_{t,j} | Per-head weight: raw linear . H_I^{-1/2} (no softmax) | |

> The small/nano configs in this repo deliberately shrink H_I and D_I for single-GPU
> training; the scoring head matches transformers' `DeepseekV4IndexerScorer`
> (`sum_h w_{t,h}.ReLU(q_{t,h}.k_s).D_I^{-1/2}`) exactly.

### DeepSeek V3 baseline included

- **MLA**: Low-rank KV compression (`c^K = kv_a_proj(h)`), shared across all query heads (MQA mode)
- **Group-limited MoE Router**: Sigmoid gating, per-group top-k selection, FP32 e-score correction bias
- **Shared expert branch**: Always-active expert added in parallel to routed experts
- **YaRN RoPE**: Interleaved positional encoding with long-context scaling (the
  Lightning Indexer shares the same YaRN frequencies; see Implementation Notes)

### Two-stage training recipe

| Stage | Steps (paper) | What's trained | Loss objective | Attention mode |
|-------|-------|---------------|----------------|----------------|
| 1: Dense warm-up | 1 000 | Indexer only (rest frozen) | KL(p_attn \|\| q_indexer) | Dense (full) |
| 2: Sparse training | 15 000 | All parameters | LM + lambda.KL | Sparse (top-k) |

Stage 1 aligns the Lightning Indexer to the main attention distribution before
enabling sparsity. Without this, the indexer selects poorly and perplexity degrades.

**Gradient isolation (paper Sec. 2.1).** The indexer input is detached from the autograd
graph, so the KL term updates *only* the indexer and the LM loss updates *only* the
main model: even though they are summed into one objective. This is enforced in
`DeepSeekV32Attention` (the indexer is fed `hidden_states.detach()`) and verified by
`tests/test_depth.py::test_gradient_isolation_kl_does_not_touch_main_model`.

### The better test: Pre trained => DSA fine-tuning

The paper's exact methodology at small scale: start from a converged dense
model, add DSA via continued fine tuning, measure validation PPL at each stage.

```bash
python3 scripts/proof_dsa_from_pretrained.py \
    --checkpoint output/wikitext_comparison/dense/checkpoint-final/checkpoint.pt \
    --warmup-steps 150 --sparse-steps 300
```

All four rows below are evaluated in **sparse mode** (the indexer is in the eval
path) except the dense rows. The honest comparison is the last one: DSA vs a
**compute-matched** dense model that received the *same* number of extra gradient
steps: so any difference is attributable to sparsity, not to training longer.

| Stage | Val PPL | Note |
|-------|---------|------|
| Dense baseline (pre-trained, 400 steps on WikiText-2) | 365.67 | reference |
| Load into DSA model, **random indexer**, sparse eval | 404.05 | +10%  naive sparsity hurts |
| After 150-step warm-up (indexer aligned), **sparse eval** | 369.38 | +1.0%: alignment nearly recovers dense quality |
| DSA: warm-up + 300 sparse steps (top-k=128, 25% context) | **322.11** | |
| **Dense + 300 steps (compute-matched baseline)** | 322.10 | fair comparison target |
| **Delta DSA vs compute-matched dense** | **+0.00%** | [OK] equal quality at 25% attention context |

What this shows (numbers regenerated with the corrected indexer; reproduce with the
command above):

1. **Naive sparsity hurts** (+10%): a *random* indexer selecting 25% of tokens
   degrades quality. The Lightning Indexer is not optional.

2. **The warm-up does real work** (+1.0% in **sparse** eval): aligning the indexer
   to dense attention via the KL loss: with the main model frozen: lets the
   *sparse* model nearly match the dense baseline before any sparse fine-tuning.
   (Evaluating the warm-up in *dense* mode would be meaningless: dense attention
   never invokes the indexer, so it trivially equals the dense baseline regardless
   of what the indexer learned.)

3. **DSA preserves quality at equal compute** (+0.00%, within noise): against a
   dense model trained for the *same* number of steps, DSA matches PPL while
   attending to only 25% of the context. This is the paper's claim: *efficiency
   without quality loss*: not an "outperforms dense" claim, which at this scale
   would just be measuring extra training steps.

```bash
python3 scripts/ablations.py            # A1-A4
```

### A1: Does the indexer recover what dense attention attends to? (the core proof)

The paper trains the indexer (Eq. 2/3) to predict which tokens dense attention
focuses on. So measure exactly that: on held-out text, for each query take dense
attention's top-k keys and ask what fraction the indexer's top-k recovers
(**recall**), and how much of dense attention's probability mass the indexer's
selection retains (**mass**). k=128 of 512 (25% context).

| Selector | recall@128 | attn-mass@128 |
|----------|-----------|---------------|
| **Lightning Indexer (warmed up)** | **0.770** | **0.676** |
| Random indexer (pre-warmup) | 0.461 | 0.463 |
| Random selection | 0.461 | 0.462 |
| Stride selection | 0.444 | 0.410 |

[OK] After warm-up the indexer recovers **77% of dense attention's chosen tokens and
68% of its probability mass**, far above the ~46% chance level that random/stride
(and the untrained indexer) achieve. This is the most direct evidence the indexer
learns its objective: it's *measured*, not inferred from downstream loss.

### A2: Learned indexer vs naive sparsity (compute-matched, real PPL)

All four start from the dense checkpoint and get the same number of fine-tuning
steps; the only difference is *how* the 25% of tokens are chosen.

| Variant | Token selection | Val PPL | Delta vs Dense |
|---------|-----------------|---------|-----------|
| Dense | full attention | 323.9 | |
| **DSA-Learned** | Lightning Indexer | **322.1** | **-0.5%** [OK] |
| DSA-Random | random positions | 346.1 | +6.9% |
| DSA-Stride | uniform stride | 363.0 | +12.1% |

[OK] Clean ordering **Learned << Random << Stride**: the learned indexer matches dense
quality at 25% context, while naive selection costs 7-12%. The indexer does real work.

### A3: top-k sensitivity (quality vs sparsity)

Warm up once, then evaluate at several k (no per-k retraining):

| top-k | % context | Val PPL |
|-------|-----------|---------|
| 32 | 6% | 407.7 |
| 64 | 12% | 382.0 |
| 128 | 25% | 370.1 |
| 256 | 50% | 366.2 |
| dense | 100% | 365.7 |

[OK] Graceful, monotone degradation: quality trades smoothly against sparsity and
approaches dense as k grows, with no cliff.

### A4: Attention FLOP scaling: O(L^2) -> O(L.k)

Architecture-exact attention FLOPs (k=128 fixed):

| seq_len | Dense GFLOP | DSA GFLOP | ratio |
|---------|------------|-----------|-------|
| 512 | 0.40 | 0.10 | **4x** |
| 2 048 | 6.44 | 0.40 | **16x** |
| 8 192 | 103.1 | 1.61 | **64x** |
| 32 768 | 1 649 | 6.44 | **256x** |
| 131 072 | 26 388 | 25.8 | **1024x** |

[OK] Reduction grows linearly with L once L > k. At the 671B model's 128K context with
k=2 048 this is a **64x** attention-FLOP reduction.

> **Note on wall-clock:** at this small hidden size DSA is *not* faster in wall time -
> the indexer + gather overhead dominates and the O(L.k) FLOP win only converts to
> latency at production scale (large hidden, long context, batched inference, fused
> FP8 kernels). The FLOP reduction is real; the wall-time win is scale-dependent.

### A5: Does the faithful (YaRN, shared frequency) indexer RoPE matter?

The official indexer reuses the main attention's YaRN scaled frequencies; a naive port
uses plain rope_theta on the indexer instead. This experiment isolates that one
variable: pretrain a small model with YaRN active on WikiText-2 past the original
window, then deep-copy it into two arms that differ *only* in the indexer frequencies,
warm up each indexer identically, and measure recall against dense attention's top-k
split by query position. Reproduce with:

```bash
python3 scripts/exp_indexer_rope.py --top-k 32
```

| indexer RoPE | recall in-window (<128) | recall extended (>=128) | warm-up KL |
|--------------|-------------------------|--------------------------|------------|
| **YaRN (faithful)** | 0.878 | **0.789** | 0.0439 |
| plain rope_theta (naive) | 0.861 | 0.750 | 0.0509 |
| **gain** | +0.017 | **+0.039** | lower KL |

[OK] The faithful indexer recovers more of dense attention's chosen tokens and aligns
to a lower KL. The signature of the fix: the gain more than doubles in the extended
region (+0.039 vs +0.017), exactly where plain frequencies diverge from the YaRN
shaped attention the indexer is trying to match.

### Summary: thesis claims

| Claim | Experiment | Result |
|-------|-----------|--------|
| Indexer learns to mimic dense attention | A1 | [OK] 0.77 recall / 0.68 mass vs 0.46 chance |
| Indexer beats naive sparse selection | A2 | [OK] Learned -0.5% << Random +6.9% << Stride +12.1% |
| Preserves quality at equal compute | Definitive Proof | [OK] +0.00% vs compute-matched dense at 25% context |
| Reduces complexity O(L^2) -> O(L.k) | A4 | [OK] 64x at 8K, 1024x at 128K |

## Implementation Notes

**KV cache** stores `(c^K, k^R, k^I)` per layer: the compressed MLA latent `c^K`
(kv_lora_rank-dim) and rope key `k^R` (never the expanded K/V), plus the Lightning
Indexer's RoPE'd key `k^I` (indexer_dim). The indexer key is projected from the raw
hidden state `h_s` (paper Eq. 1), so it must be cached separately: the raw hidden
states are not retained. New query tokens are scored against the full `k^I` cache,
then MLA attention runs on the gathered top-k entries only.

**Indexer RoPE** (faithful to the official `DeepSeek-V3.2-Exp/inference/model.py`):
the Lightning Indexer is position-aware in the same three ways the official code is.
(1) It reuses the *same* YaRN-scaled frequencies as the main attention, via the shared
`compute_rope_inv_freq` helper, so its sense of position matches the attention it
imitates even in the long-context regime. (2) It uses the non-interleaved (half-split)
rotary layout even when MLA uses the interleaved layout. (3) Only the first
`qk_rope_head_dim` channels of each indexer head carry rotary position; the rest are
left unrotated (the official 128-dim head splits into 64 rope + 64 nope). The YaRN
mscale is a softmax temperature, not part of the rotary vectors, so the indexer omits
it. Verified by `tests/test_indexer_rope.py`.

**FP32 safety**: The KL loss computation always promotes to float32 regardless of
model precision. Causal masking places `-inf` in index scores; `0.(-inf)` is NaN in
bf16/fp16. A `nan_to_num` guard handles any residual masked positions.

**E-score correction bias** (MoE router): stored and computed in float32 even
during bf16 training, matching the paper's implementation.

**Indexer gradients**: top-k selection is non-differentiable, so the LM loss cannot
reach the indexer. Conversely, the indexer input is detached, so the KL loss cannot
reach the main model. The indexer is therefore trained *exclusively* via the KL loss
(stage 1) and the KL regularisation term (stage 2); the main model is trained
*exclusively* via the LM loss: matching the paper's separate-optimisation rule.

**FSDP checkpointing**: uses `StateDictType.FULL_STATE_DICT` with
`rank0_only=True`. Only rank-0 writes to disk; no manual gather needed.
